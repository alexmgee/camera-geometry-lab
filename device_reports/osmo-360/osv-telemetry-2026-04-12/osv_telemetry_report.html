Container Structure

How to read this section: The OSV file is a standard MP4 container (ISO Base Media File Format) with DJI-specific extensions. It holds two separate fisheye video streams (front and back lenses of the dual-lens camera), along with metadata tracks that use the djmd codec tag for telemetry and dbgi for debug data.

The LRF sidecar is a companion file written simultaneously. It contains a low-resolution pre-stitched equirectangular video (2048×1024, 2:1 aspect ratio) at half the framerate, plus its own telemetry tracks. This is useful for quick preview and validation without running DJI Studio's stitching pipeline.

Both files share the same creation timestamp and the same protobuf schema for their djmd tracks.

Protobuf Schema

How to read this section: DJI embeds telemetry in the djmd MP4 tracks using Google Protocol Buffers. The schema file is dvtm_oq101.proto, identified from the file's own metadata. ExifTool v13.41 has built-in support for this schema — no separate .proto file is needed.

Proto field paths are hierarchical (e.g., 3-2-9-1 means message 3, field 2, sub-field 9, component 1). Message 1 appears once per file and contains device identification. Message 3 repeats per frame and contains the telemetry data we need.

The highlighted rows in the tables below are the fields critical for the gravity-aware masking pipeline.

Quaternion Time Series

How to read these charts: A quaternion is a four-component representation of 3D rotation: (w, x, y, z). When the camera is in its reference orientation (level, facing north), the quaternion is approximately (1, 0, 0, 0). As the camera rotates, the components change smoothly.

The top chart shows all four components over the 80-second clip. Smooth, continuous curves indicate clean sensor fusion data. Sudden jumps would indicate data corruption or sensor failure — none are present. The wide sweeps across all components confirm the camera was rotated through many orientations during capture.

The norm chart shows that each quaternion has magnitude exactly 1.0, which is a mathematical requirement for valid rotation quaternions. Deviation from 1.0 would indicate data corruption. The scale is zoomed to ±0.0000002 to show the negligible float32 rounding variation.

Every 5th data point is displayed for rendering performance. The full dataset contains 4003 frames.

Euler Angles

How to read this chart: Euler angles (yaw, pitch, roll) are a more intuitive representation of the same orientation data. They are computed from the quaternions using the ZYX convention:

• Yaw (blue): horizontal heading, −180° to +180°. Changes as the camera pans left/right.
• Pitch (green): vertical tilt, −90° (straight down) to +90° (straight up).
• Roll (red): rotation around the viewing axis, −180° to +180°.

The wide ranges across all three axes confirm "alien eye" capture style — the camera was tilted, panned, and rolled through nearly every possible orientation during this 80-second clip. This is expected behavior when the operator waves the camera on a selfie stick into corners, under objects, and overhead.

Note: Euler angles suffer from gimbal lock near ±90° pitch, causing apparent jumps in yaw and roll. These are a visualization artifact, not data errors. The underlying quaternion data is smooth and continuous.

Accelerometer

How to read these charts: The accelerometer measures linear acceleration in three axes, reported in units of g (where 1 g = 9.81 m/s²). When the camera is stationary, the accelerometer reads approximately 1 g total (gravity). When in motion, the reading includes both gravity and the camera's own acceleration.

The component chart shows the X, Y, and Z readings separately. For a handheld camera, one axis will be dominated by gravity (~1 g) while the others fluctuate with motion. The dominant gravity axis shifts as the camera is tilted.

The magnitude chart shows the total vector length: √(X² + Y² + Z²). This hovers near 1.0 g during steady motion. Values significantly above 1 g indicate the camera is accelerating (e.g., being swung), while values near 0 indicate brief moments of near-freefall (e.g., the camera reaching the apex of an upward swing).

The accelerometer data confirms the IMU is actively reporting and can be used alongside the quaternions for motion characterization.

Data Quality Validation

How to read this section: Each test verifies one property that the quaternion data must satisfy for use in the gravity-aware masking pipeline. "PASS" means the measured value meets the criterion.

The frame-to-frame delta chart below the table shows the Euclidean distance between consecutive quaternions. This measures how much the camera's orientation changed between frames. For a 50 fps video, a delta of 0.01 corresponds to roughly 1° of rotation. The maximum observed delta of 0.044 corresponds to approximately 5° per frame, which is vigorous but physically plausible handheld motion.

A sudden spike to a large value (e.g., > 0.5) would indicate a data discontinuity or corruption. None were found in this dataset.

Extraction Commands

How to use this section: These are the minimal commands needed to reproduce the telemetry extraction on any DJI Osmo 360 .osv file. All require only open-source tools available on all platforms.

ExifTool's -ee flag (ExtractEmbedded) is the key — it reads per-frame timed metadata from the djmd data tracks. Without this flag, ExifTool only reads file-level metadata. The -a flag shows all instances of duplicate tags (needed since the same field appears 4003 times), and -u shows unknown tags (the unnamed proto fields).

1. Full Telemetry Dump

exiftool -ee -a -u -n file.osv

2. Quaternions + Accelerometer Only

exiftool -ee -a -u -n file.osv \
  | grep -E "^(Sample Time|Dvtm oq 101 3-2-9-[1234]|Accelerometer)"

3. Container Probing

ffprobe -v quiet -print_format json -show_format -show_streams file.osv

4. Proto Schema Identification

exiftool -a -u file.osv | grep "Category"
# pb_file:dvtm_oq101.proto;model_name:OQ001;pb_version:2.0.8

5. Extract LRF Preview Frame

ffmpeg -i file.LRF -vframes 1 -map 0:0 preview_erp.png
